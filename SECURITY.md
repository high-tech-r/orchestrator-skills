# セキュリティ対策のご説明

本プロジェクトは、AIを活用した開発において**セキュリティを自動検査の仕組みで担保**しています。
本書は、お客様に「どのようにセキュリティを確保し、どのようにテストしているか」をご説明するものです。

> 開発者・技術者向けの詳細な設定手順は [`docs/security/LEVEL2_SECURITY.md`](docs/security/LEVEL2_SECURITY.md) を参照してください。

## 基本方針

すべての変更は、人手のレビューに加えて**自動セキュリティ検査（品質ゲート）を通過しなければ反映できません**。
検査に1つでも重大な指摘があれば、その変更は自動的に差し戻されます（人の判断に依存しません）。

## 何を検査しているか

| 観点 | 内容 | 主な対応リスク |
|---|---|---|
| 秘密情報の漏洩 | APIキー・パスワード等がコードに混入していないか | 認証情報の流出 |
| 既知の脆弱性 | 利用しているライブラリに既知の脆弱性がないか | OWASP A06:2021 脆弱なコンポーネント |
| 不正な依存（AI起因） | AIが提案した実在しない／なりすましパッケージの混入 | サプライチェーン攻撃 |
| ソースコードの脆弱性 | SQLインジェクション・XSS 等の典型的な欠陥 | OWASP Top 10 / CWE Top 25 |
| AI生成コード特有のリスク | プロンプトインジェクション・未承認のAI利用 | AI駆動開発に固有の脅威 |
| ライセンス | オープンソースのライセンス違反がないか | 法的リスク |
| 実行時の脆弱性 | 動作中アプリへの外部からの攻撃耐性（DAST） | 認証・認可・設定不備 |

## どのようにテストしているか

1. **コミット前** — 開発者の手元で秘密情報スキャンが走り、漏洩を未然にブロックします。
2. **変更ごと（CI）** — 上表の検査を自動実行。重大な指摘があればマージを自動拒否します。
3. **実装の誠実性チェック** — 「成功に見えて実際は動作しない」コード（例: 通知を送らずに『送信しました』と返す、
   エラーを隠してログイン状態に見せる等）を専用の検査で検出し、差し戻します。
   *見た目の動作だけでなく、実際の処理結果まで検証します。*
4. **動的テスト（DAST）** — 必要に応じて、実際に起動したアプリへ外部からリクエストを送り、実行時の脆弱性を確認します。

## 証跡（エビデンス）の提示

検査のたびに、以下を**ダウンロード可能な記録として保存**しています。ご要望に応じて提出可能です。

- **スキャン結果**（脆弱性・コード解析の指摘一覧）
- **SBOM（ソフトウェア部品表 / CycloneDX形式）** — 使用している全コンポーネントの一覧
- **ライセンスレポート** — 各コンポーネントのライセンス一覧
- **検査の合否記録** — 各変更が品質ゲートを通過したことの記録

## 使用ツール（すべて実績あるツール）

Gitleaks / TruffleHog（秘密情報）、Trivy / OSV-Scanner（脆弱性・SBOM・ライセンス）、
Semgrep（静的解析・AI生成コード対策）、OWASP ZAP（動的解析）。
業界標準のオープンソースツールで構成し、特定ベンダーへの依存を避けています。

## 脆弱性の報告

セキュリティ上の問題を発見された場合は、公開Issueではなく、プロジェクト管理者まで直接ご連絡ください。
