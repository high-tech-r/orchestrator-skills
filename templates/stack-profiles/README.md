# templates/stack-profiles/ — スタックプロファイル

フレームワークを**言語/フレームワークに依らず**動かすための仕組み。

- **フレームワーク本体が所有するのは「観点（契約）」** … `_template.md` の見出し
  （起動/テスト/種別ごとの確認/テストダブル/依存/落とし穴）。これは普遍で腐らない。
- **具体（コマンドやイディオム）はプロファイルに置く** … `laravel.md` / `fastapi.md` は
  **リファレンス例**（権威ではない。バージョンで腐る前提・「要確認」を明記）。

## どう使われるか

`orchestrate` は初期化時、適用先プロジェクトに**アクティブなプロファイル**が無ければ:
1. `tech_stack` に一致する同梱例があれば、それを下敷きにする
2. 無ければ `_template.md` の契約に沿って、**実スタック用のプロファイルを生成**する
3. 生成物は**適用先プロジェクト**（`.orchestrator/stack-profile.md`）に保存する

以後 `design` / `test-design` / `implement` / `review-guide` は、この**アクティブなプロファイル**を
参照して、起動・動作確認・テスト・テストダブルの具体を決める。

> 同梱例が古くても害は小さい（権威ではないため）。適用先の Claude Code が現行バージョンと
> 食い違いに気づいたら、プロジェクト側のプロファイルを更新する ── これが本仕組みの前提。

## 新しいスタックに対応する

`_template.md` をコピーして全見出しを埋めるだけ。例として残すなら本フォルダに、
プロジェクト固有なら `.orchestrator/stack-profile.md` に置く。
