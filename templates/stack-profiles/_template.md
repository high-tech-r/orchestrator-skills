# スタックプロファイル契約（テンプレート）

スタックプロファイルは、特定の言語/フレームワークで「どう動かし・どう確認し・どうテストするか」を
1ファイルに記述したもの。**フレームワーク本体は『観点（この契約）』を所有し、具体は各プロファイルに置く。**

新しいスタックに対応するには、このテンプレートをコピーして以下の見出しをすべて埋める。
`orchestrate` は適用先プロジェクトにプロファイルが無ければ、この契約に沿って**実スタック用を生成**する
（同梱の `laravel.md` / `fastapi.md` は権威ではなく**リファレンス例**。最新バージョンで要確認）。

---

## メタ情報
- 対象スタック: （例: Laravel 11 / PHP 8.3）
- 最終確認日: YYYY-MM-DD
- 出典: （公式ドキュメント等。腐敗検知の手がかり）

> 適用先の Claude Code へ: 下記コマンド/イディオムが現行バージョンと食い違う場合は、
> 古い可能性が高い。鵜呑みにせず、実環境で確認して更新すること。

## 1. 起動・動作確認
- アプリの起動コマンド（Docker前提。コンテナ内/外どちらで何を実行するか）
- 「生きている」ことの確認方法（ヘルスURL / コマンド / ログ）

## 2. テスト実行
- テストの実行コマンド（フレームワーク標準のもの）
- カバレッジの取得方法（あれば）

## 3. インターフェース種別ごとの動作確認
機能の種別ごとに「手動でどう叩く/確認するか」を書く。該当するものだけでよい。
- HTTP API: （curl / APIドキュメントUI 等）
- 非同期ジョブ / キュー: （投入方法・ワーカー起動・結果確認）
- イベント / リスナー: （発火方法・確認）
- CLIコマンド: （実行方法）
- スケジュール: （手動トリガ方法）

## 4. テストダブル（外部作用の検証）
「成功に見えて実は動いていない」を防ぐため、外部作用は **fake/mock で『呼ばれたこと』まで検証**する。
スタック標準のイディオムを書く。
- メール送信:
- キュー / ジョブ投入:
- 通知:
- イベント:
- 外部HTTP:
- ストレージ / ファイル:

## 5. 依存・必要サービス
- 依存定義ファイル（例: composer.json / requirements.txt / package.json）
- 動作に必要な周辺サービス（DB / キュー / メールキャッチャ 等）と起動方法

## 6. よくある落とし穴・トラブルシュート
- 環境設定のハマりどころ（.env のキー、マイグレーション、キュー接続 等）
- 典型的なエラーと対処
