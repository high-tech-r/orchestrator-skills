# Skill: consistency-check（整合性チェック）

## いつ使うか
- Gate 1: 設計書の品質チェック（設計完了後）
- Gate 2: 設計書⇔コード⇔テストの整合性チェック（実装＋テスト完了後）

## Gate 1: 設計レビュー

### 入力
- `docs/design/F-XXX.md`
- バックログの該当機能の受入条件

### チェック項目
| チェック | 判定方法 | NG時 |
|---------|---------|------|
| 要件カバレッジ | 受入条件の全項目が設計に反映されているか | 不足箇所を指摘 |
| 仕様の矛盾 | 設計書内で矛盾する記述がないか | 矛盾箇所を指摘 |
| 曖昧性 | 「適宜」「必要に応じて」等の曖昧表現 | 具体化を指示 |
| トレーサビリティ | 受入条件トレーサビリティの参照先が実在し内容が一致するか | 対応不十分な箇所を指摘 |
| エラーカテゴリ | 4カテゴリ（バリデーション/ビジネスロジック/外部サービス/インフラ）が検討されているか | 不足カテゴリを指摘 |

### 出力
`docs/consistency_report/F-XXX_gate1.md`

## Gate 2: 整合性チェック

### 入力
- `docs/design/F-XXX.md`（設計書）
- `src/` の該当コード
- `docs/test_spec/F-XXX.md`（テスト仕様書）
- `tests/` の該当テストコード

### チェック項目
| チェック | 判定方法 | NG時 |
|---------|---------|------|
| 機能実装カバレッジ | 設計書の全機能がコードに実装されているか | 漏れを指摘 |
| 余剰コード | 設計書にない処理がコードにないか | 不要なら削除を指示 |
| API仕様一致 | エンドポイント・リクエスト・レスポンスが設計書通りか | 乖離を指摘 |
| DB定義一致 | テーブル定義が設計書と一致しているか | 乖離を指摘 |
| テストカバレッジ | テスト仕様書が全受入条件をカバーしているか | 不足を指摘 |
| 用語統一 | 変数名・メソッド名・テーブル名が設計書と一致 | 不一致を指摘 |
| 依存パッケージ整合 | コードでimport/requireしているパッケージが全て依存定義ファイル（requirements.txt / package.json / go.mod 等）に記載されているか | 不足を指摘 |
| Docker実行可能性 | Dockerfile + docker-compose.yamlでビルド・起動できる構成か | 不備を指摘 |
| Tier判定妥当性 | `review_tier_definition.yaml` のkeywords/risk_signalsと実装コードを照合し、バックログのTier指定が妥当か検証。決済・個人情報・認証を扱う機能が Tier C/D（低リスク）に設定されていないか | Tier引き上げを推奨 |
| 偽の成功（最優先） | 副作用（メール送信・DB保存・状態変更）を伴わずに成功（200/201・成功メッセージ）を返していないか。未実装処理を成功扱いしていないか。**失敗時のフォールバックが「成功」として報告されていないか**（例: 処理失敗時に元データを返して completed 扱い） | implementに差し戻し（blocking） |
| エラーの握り潰し | 空の`catch`・例外を飲んで`null`/`true`を返す箇所がないか。特に401で握り潰してリカバリしていない箇所 | implementに差し戻し（blocking） |
| 壊れたデータ生成 | 必須カラム欠落のレコード作成（例: `hashed_password` NULL）や、異常系での勝手な補完生成がないか | implementに差し戻し（blocking） |
| 無効値の送信 | `Bearer null` 等、前提が揃わないまま無効リクエストを投げて失敗を曖昧にしていないか | implementに差し戻し（blocking） |
| 境界契約の突合 | この機能が跨る境界（サーバー設定⇔クライアント実装、サービス間API、同一リソースを更新する他経路）を列挙し、**相手側の実装を実際に開いて**前提が一致するか確認したか。既存機能と共有する状態（例: 注文ステータス）を更新する場合、他経路との競合を確認したか | 乖離箇所を指摘（前提を破っている側に修正指示） |
| 並行性・冪等性 | 状態の確認と変更が同一ロック/トランザクション内か（check-then-act の分離がないか）。Webhook/イベントハンドラは現在状態を確認してから遷移し、同一イベント2回で安全（冪等）か | implementに差し戻し（blocking） |

### 「嘘をつくコード」の検出手順（最優先・CLAUDE.md実装の誠実性に対応）
上記4項目は実害が大きく、人間テスターが「成功」と誤認するため、Gate 2 で必ず精査する。

- **副作用の追跡**: 成功を返すエンドポイント／関数について、設計書が要求する副作用（送信・保存・更新）が
  コード上で実際に実行されているかを行単位で確認する。「成功メッセージを返すだけ」を検出したら blocking。
- **例外ハンドラの精査**: `catch`/`except` ブロックを全て確認し、ログも再送出も状態通知も無い握り潰しを検出する。
- **生成データの妥当性**: `INSERT`/`create` の直前で、必須・NOT NULL 制約や後続機能（ログイン等）が成立する値が
  揃っているかを確認する。
- **境界契約の突合**: 機能が跨る境界（設定⇔クライアント、サービス間API、同一状態を触る他経路）を列挙し、
  **相手側のコード/設定を実際に開いて**1対ずつ前提の一致を確認する。片側だけ読んで「たぶん合っている」は禁止。
- **並行性・冪等性**: check-then-act の分離（確認→ロック外で遷移）は逐次実行のテストでは絶対に検出できないため、
  コードを読んで判定する。「失敗したら元データで続行」のフォールバックも、テストも実行も正常に見えるまま
  壊れた成果物を成功として納品する偽成功の亜種として精査する。
- これらは **blocking（重要度 high）** とし、1件でもあれば FAIL。テストが通っていても見逃さない
  （偽の成功はテストも「成功」を返すため、テスト緑＝安全ではない）。

なお Gate 2 は**実装中の1機能**に閉じた縦のチェックであり、機能をまたぐ横断的欠陥（同型 IDOR の量産、
機能間の状態競合など）はオンデマンドの `audit` スキルが全域横断で補完する（二層防御）。

### テスト品質の検出基準（共通ルールを参照）
テストコードの品質は **`.claude/skills/_shared/test-quality-rules.md`（共通テスト品質ルールの真実源）** に照らして検証する。
特に以下の偽陽性アサーションを検出したら test-design / implement に差し戻す:
- `assertNull` をキー存在確認（`assertArrayHasKey`）なしで使い、キー不在を見逃している箇所
- URL フィールドを `assertNotNull` だけで検証し、パス形式（含有）まで確認していない箇所
- 正常系 API テストでフィールド完全一致チェック（`assertEqualsCanonicalizing` 等）を欠き、内部フィールド漏洩を見逃す箇所

### 出力
`docs/consistency_report/F-XXX_gate2.md`

## 出力フォーマット（共通）

```markdown
# 整合性チェックレポート: F-XXX（Gate 1/Gate 2）

## 結果: PASS / FAIL

## チェック結果
| チェック項目 | 結果 | 詳細 | 重要度 | 修正先 |
|------------|------|------|--------|--------|
| 要件カバレッジ | ✅ PASS | ... | - | - |
| 曖昧性 | ❌ FAIL | 「適宜対応」が3箇所 | medium | design |

## サマリー
- 合計: N件
- PASS: N件
- FAIL: N件
- ブロッキング: Yes/No

## 推奨アクション
- designに差し戻し: 〇〇を修正
```

## 重要なルール
- FAILの場合、具体的にどこをどう直すかを明記する（「問題あり」だけでは不十分）
- remediation_target（修正先: design / implement / test-design）を必ず指定
- 3回FAILが続いたらユーザーにエスカレーション。AIでは解決できない可能性を伝える
