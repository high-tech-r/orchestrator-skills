# Skill: quality-report（顧客向け品質レポート生成）

## いつ使うか
- リリース・納品・定期報告時に、品質状況（テスト・バグ・ゲート通過）を顧客に示すとき
- 「品質レポートを作って」と指示されたとき

機能単位のパイプライン外で、求められたときに単発実行する。
セキュリティ面は `security-report`、本スキルは**機能品質**を担当する。

## 入力（実データから転記。推測しない）
- `docs/backlog.md` … 機能一覧・Tier・ステータス
- `docs/test_spec/F-XXX.md` … テストケース数・受入条件カバレッジ
- `tests/` のテスト実行結果（`docker compose exec app <テストコマンド>` の出力）
- `docs/consistency_report/F-XXX_gate1.md` / `_gate2.md` … Gate 1/2 の合否・指摘
- カバレッジ計測があればその数値

## 出力ファイル
`docs/delivery/quality_report_YYYY-MM-DD.md`（顧客が読める日本語・実日付）。

```markdown
# 品質レポート

- 対象: （プロジェクト名 / リリース or コミットSHA）
- 作成日: YYYY-MM-DD

## 1. 総合判定
✅ 全機能が品質ゲート（Gate 1/2）を通過 / ⚠️ 未通過・既知の問題あり

## 2. 機能別サマリー
| 機能 | Tier | テスト件数 | 合否 | Gate1 | Gate2 | 受入条件カバレッジ |
|------|------|-----------|------|-------|-------|------------------|
| F-001 タスク追加 | B | 8 | ✅ 8/8 | ✅ | ✅ | 3/3 |

## 3. テスト結果
- 総テスト件数 / PASS / FAIL / SKIP
- カバレッジ（計測していれば。Tierごとの基準: S=90 / A=80 / B=60 …）
- 計測していない場合は「未計測」と明記

## 4. 既知の不具合・残課題
| # | 内容 | 影響 | 重大度 | 状態（対応済/対応予定/許容） |
（無ければ「現時点で既知の不具合なし」と明記）

## 5. 受入条件トレーサビリティ
全受入条件が、どのテストでカバーされているかの対応表（未カバーがあれば明示）。

## 6. テスト方法の説明
どのようにテストしたか（自動テスト＋手動レビュー手順書 `docs/review_map/`）を顧客向けに1段落で。
```

## 重要なルール（誠実性 — CLAUDE.md実装の誠実性に対応）
- **FAIL・未計測・未対応を隠さない。** 「テスト未実施」「カバレッジ未計測」はそのまま書く。
- テスト件数・合否は実行結果から転記する。緑に見せるための丸めをしない。
- **「テストが緑＝バグなし」とは書かない。** カバーした範囲を明示し、範囲外は範囲外と書く。
- 既知の不具合を「軽微」と勝手に格下げしない。重大度は事実ベースで。
- 表現は `SECURITY.md` / `docs/delivery/index.md` と整合させる。
