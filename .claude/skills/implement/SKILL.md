# Skill: implement（ソースコード生成）

## いつ使うか
- 設計書に基づいてソースコードを生成するとき
- Gate 2で差し戻されてコードを修正するとき
- テスト仕様書に基づいてテストコードを生成するとき

## 入力
- `docs/design/F-XXX.md` の設計書
- 技術スタック情報（project_status.yamlから）
- 既存コード（追加開発の場合）
- テストコード生成時: `docs/test_spec/F-XXX.md` + 実装済みコード

## 出力
- ソースコードを `src/` 以下に直接ファイルとして作成
- テストコードを `tests/` 以下に直接ファイルとして作成

## コーディングルール
- 設計書のAPI仕様・DB定義に厳密に従う
- 変数名・メソッド名・テーブル名は設計書の用語と統一する
- 型ヒントを付ける（Python）/ 型定義を付ける（TypeScript）
- 各ファイルの先頭にdocstringで「この機能のどの部分を実装しているか」を記載

## 実装の誠実性（必須・最優先 — CLAUDE.mdの最優先原則を実装で守る）
「動いているように見えて実は壊れている」コードを**絶対に生成しない**。以下は禁止事項。

1. **偽の成功を返さない**
   - 未実装の処理を成功扱いにしない。実装できないなら `501`/`503` を返すか、明示的に未実装とわかる形にする
   - 成功メッセージ・成功ステータス（200/201）は、**実際に副作用（メール送信・DB保存・状態変更）が完了した場合のみ**返す
2. **例外を無音で握り潰さない**
   - 空の `catch`／例外を飲んで `null`・`true` を返す実装は禁止。失敗は呼び出し元に伝播させる（エラー状態 or 例外 or `false`）
   - **401（認証エラー）は必ずリカバリ**（ログアウト→ログイン誘導）。握り潰して正常画面に留めない
3. **壊れたデータを作らない**
   - 必須カラムが欠けたレコード（例: `hashed_password` NULL）を作成しない。作成前に後続機能が成立するか確認する
   - 既存ユーザーが居ない等の異常系は、勝手に補完生成せず適切なエラー（例: `404`）を返す
4. **無効な値を送って失敗を隠さない**
   - トークンが `null` のまま `Authorization: Bearer null` を送る等、無効なリクエストを投げて失敗を曖昧にしない。前提が揃わない場合は呼ばずに未充足を明示する
5. **フォールバックを偽成功にしない**
   - 失敗時に代替値・元データで処理を続行する場合、必ず (a) ERROR ログ、(b) 呼び出し元に伝わるフラグ/warning をレスポンスに含める。無言のフォールバックは偽成功と同罪（例: 加工失敗時に元データを返して completed と報告）
   - 外部から取得した値（HTTPレスポンス等）は検証してから使う（ステータスコード・Content-Type・パース可能性）。エラーページを「成功したデータ」として返さない
6. **配線まで完結させる**
   - コールバック・フック・シグナルを宣言したら、**代入（登録）箇所まで同時に実装**する。「後で配線する前提の宣言」を残さない（宣言だけのコールバックは監査まで誰も気づかない）
   - パラメータを受け取るなら処理に反映する。反映できないなら受け取らない（no-op パラメータ＝嘘のインターフェース禁止）
7. **キャッシュキーには「変わり得る変数」を全て含める**
   - ID だけをキーにして中身（URL・バージョン）の変化を無視しない。再生成・更新後に古い内容を出し続けるのは偽成功の一種
8. **境界の相手側を読んでから実装する**
   - サーバー設定・外部APIの仕様に依存するクライアント処理（トークン保存、レスポンスのパース等）は、**相手側の設定/実装を実際に開いて**前提（ローテーションの有無、フィールド形状）を確認してから書く

これらは `consistency-check` Gate 2 と `audit` スキルで検出・差し戻しされる。生成段階で作り込まないこと。

## 依存パッケージ管理（必須）
- 新しいライブラリを使う場合、**必ず** requirements.txt（Python）/ package.json（Node.js）に追加する
- バージョンは範囲指定で固定する（例: `pydantic>=2.0.0,<3.0.0`）
- dev用のパッケージ（テスト用等）は requirements-dev.txt / devDependencies に追加
- コード内で使うライブラリが依存定義ファイル（requirements.txt / package.json / go.mod 等）に記載されていない状態を**絶対に作らない**
- Dockerfileの再ビルドで依存が解決される状態を常に維持する

### slopsquatting対策（必須・レベル2セキュリティ）
AIが幻覚した実在しないパッケージ／タイポスクワッティングを混入させない。

**依存を追加する前に Context7 MCP で「実在性」と「最新API」を照合する（ベストエフォート・助言的）:**
Context7 はハードゲートではない加算レイヤー。使えるときは上乗せの保護、使えないときは
従来ルールにフォールバックする。ただし**劣化は必ず可視化する**（下記3）。

1. 依存定義ファイル（requirements.txt / package.json / go.mod 等）に新規パッケージを
   追加する前に、Context7 の `resolve-library-id` でライブラリ名を解決する。
   - **resolve できた** → 実在の肯定的証拠として扱ってよい。
   - **resolve できない** → 自動却下はしない。**「要確認」フラグを立ててユーザーに確認を促す**
     （Context7 は*ドキュメント化された*ライブラリのインデックスであり、実在するがマイナーな
     パッケージは未収録のことがある。解決不能は「疑わしい」であって「存在しない」ではない）。
2. そのライブラリで使うAPI（クラス・関数・引数・import パス）は `query-docs` で
   **最新ドキュメントを引いて確認**してから書く。学習データが古い／幻覚のAPIをそのまま
   書かない（存在しないメソッドを呼んで握り潰す = 偽成功の温床）。
3. Context7 が使えない場合（ヘッドレス実行・MCP未接続・権限なし・**月間クォータ超過**等）は
   従来ルールにフォールバックする。このとき**「Context7 未照合」を必ず二重に可視化する**:
   - 依存定義ファイルのコメントに「Context7 未照合・要レビュー」を残す
   - **その実行のユーザー向け報告にも「未照合」と明示する**（照合済みと誤認させない。
     「照合したことにする」のは偽成功＝最優先原則違反）
   - 無認証だと月間クォータが小さく超過しやすい。フレームワークで Context7 を常用するなら
     https://context7.com/dashboard の無料 API キーを設定しておくと安定する（任意の強化オプション）。
4. **守秘**: Context7 は外部APIにクエリを送る。**送るのはライブラリ名など一般的情報に限定**し、
   顧客の独自コード・システム構成・機密情報を query に含めない（第三者への流出防止）。

従来ルール（Context7 の可否に関わらず常に適用）:
- **実在性が確認できないパッケージは追加しない。** 標準的で広く使われているパッケージのみを使う。
- 一般的でないパッケージ名を使う場合、依存定義ファイルのコメントで「各レジストリ（PyPI / npm 等）で実在・正規性を確認すべき」旨を明記し、ユーザーにレビューを促す
- バージョンを固定し、差分がレビューで明確に出る状態を保つ（CIの Trivy / pip-audit / OSV-Scanner が照合する）
- 詳細: `docs/security/LEVEL2_SECURITY.md`

## 出力後の記録
コード生成後、以下のメタ情報を `docs/design/F-XXX.md` の末尾に追記する:

```markdown
## 実装マッピング（自動追記）
| 設計の参照先 | 実装ファイル | 概要 |
|------------|------------|------|
| API仕様: POST /api/xxx | src/api/xxx.py | APIビュー |
| DB変更: xxxテーブル | src/models/xxx.py | モデル定義 |
| DB変更: マイグレーション | migrations/xxx.py | マイグレーション |
```

## テストコード生成
テスト仕様書（`docs/test_spec/F-XXX.md`）と実装済みコードの両方を読んで、
実行可能なテストコードを生成する。

外部作用（メール・キュー・通知・外部HTTP・ストレージ）の検証は、**アクティブなスタックプロファイル
（`.orchestrator/stack-profile.md`）のテストダブルのイディオム**に従う
（例: Laravel `Mail::fake()`/`Queue::fake()`、Python `unittest.mock`/`respx`）。
「呼ばれたこと」まで assert し、未実装なら「呼ばれない／501」を固定する。

- テスト仕様書のテストケースIDをテスト関数名に含める（例: `test_TC_F001_001_normal_registration`）
- モック対象は実装コードのimportパスに基づいて正確に指定する
- テスト仕様書に記載のないテストは追加しない（スコープを守る）

### テストコードの品質ルール
**テストコード生成前に `.claude/skills/_shared/test-quality-rules.md`（共通テスト品質ルールの真実源）を必ず読み**、適用する。
横断ルール（偽陽性アサーション禁止 ルール1〜3 / 全フィールド一致 / 副作用検証）はそのファイルが正。要点のみ再掲する:
- **全フィールド一致**: APIレスポンスは `set(body.keys()) == {全フィールド}` で検証。`"field" in body` の部分一致は禁止。
- **偽陽性アサーション禁止**: ①`assertNull`は`assertArrayHasKey`でキー存在確認と分離 ②URLは`assertStringContainsString`でパス形式検証（`assertNotNull`不可） ③全正常系で`assertEqualsCanonicalizing([...], array_keys($body))`のフィールド完全一致を必須化（一覧APIは各要素にも適用）。
- **副作用検証**: 成功（200/201）を返すなら副作用（メール送信・DB保存・状態変更）が呼ばれたことまでassertする。未実装なら「呼ばれない／501」を固定。
- イディオム（アサーション/テストダブル）はスタックプロファイル（`.orchestrator/stack-profile.md`）の書式に読み替える。Laravel/PHPUnit の例は真実源を参照。

このSkill固有のルール:
- **テストデータはハードコードする。** ランダムや動的生成ではなく、固定値を使って再現性を保証する。
- **1テスト関数で1テストケースを検証する。** 複数ケースを1関数にまとめない。

## 差し戻し時のルール
- feedback_historyの過去の指摘を確認し、同じ問題を繰り返さない
- 修正箇所だけでなく、修正による影響範囲も確認する
