# Skill: test-design（テスト仕様書生成）

## いつ使うか
- 設計書に基づいてテスト仕様書を生成するとき
- Gate 2で差し戻されてテスト仕様書を修正するとき

## 入力
- `docs/design/F-XXX.md` の設計書（実装コードは見ない）
- Tier情報（バックログから）

## 出力ファイル
`docs/test_spec/F-XXX.md` に以下の構造で保存する。

```markdown
# テスト仕様書: F-XXX（機能名）

## Tier: B
## テスト方針: 正常系・主要異常系・境界値・主要インフラエラーをカバー

## テストケース

### 正常系
| ID | テスト名 | 事前条件 | 入力 | 期待結果 | 優先度 |
|----|---------|---------|------|---------|--------|
| TC-F001-001 | ... | ... | ... | ... | 高 |

### 異常系
| ID | テスト名 | 事前条件 | 入力 | 期待結果 | 優先度 |
|----|---------|---------|------|---------|--------|
| TC-F001-002 | ... | ... | ... | ... | 高 |

### 境界値
（同上のテーブル形式）

### セキュリティ
（Tier S/A の場合はさらに厚くする）

### インフラ起因エラー
（外部サービス障害、DB障害等）

## 受入条件カバレッジ
| 受入条件 | カバーするテストケース |
|---------|---------------------|
| "条件1" | TC-F001-001 |
| "条件2" | TC-F001-002, TC-F001-003 |

## エラーカテゴリチェック
- [x] バリデーションエラー: TC-F001-00x
- [x] ビジネスロジックエラー: TC-F001-00x
- [ ] 認証・認可エラー: 対象外（理由: ...）
- [x] 外部サービス障害: TC-F001-00x
- [x] インフラ障害: TC-F001-00x
- [ ] タイムアウト: Tier B のため省略
```

## 重要なルール
- このSkillはテスト仕様書（何をテストするか）のみ出力する
- テストコード（どうテストするか）は implement Skill が生成する
- 実装コードは参照しない（テストファースト思想）
- 全受入条件がいずれかのテストケースでカバーされていること
- エラーカテゴリチェックリストを必ず含めること

## テスト設計の品質ルール

### レスポンス検証は「全フィールド一致」で書く
APIレスポンスのテストケースを設計する際、期待結果には**レスポンスに含まれるべき全フィールド名を明記**する。
「idが含まれている」「titleが含まれている」ではなく「レスポンスのフィールドは id, title, description, is_completed, created_at の5つであること」と書く。
これにより、フィールドの追加・削除・名前変更があった場合にテストが落ちて検出できる。

悪い例: 「レスポンスにidとtitleが含まれている」→ 余計なフィールドが増えても気づかない
良い例: 「レスポンスのフィールドは {id, title, description, is_completed, created_at} と完全一致する」

### 偽陽性を生むアサーションを禁止する（F-031 自己レビューで発見）
テストが「PASSしているのに実は何も保証していない」偽陽性を作らない。以下3点を全テストケースの期待結果に反映する。
**アサーションのイディオムは言語/FWで異なる**（下記は Laravel/PHPUnit の例）。アクティブなスタックプロファイル
（`.orchestrator/stack-profile.md`）の対応する書き方に読み替えること。

- **ルール1: null チェックはキー存在確認と分離する。**
  `$res->json('key')` はキー不在でも `null` を返すため、`assertNull` だけでは「キーが存在して null」と「キーが存在しない」を区別できない。
  必ず `assertArrayHasKey` でキー存在を先に確認してから値を検証する。
  - 期待結果の書き方: 「`key` フィールドが存在し（キー存在を確認）、かつ値が null であること」

- **ルール2: URL フィールドはパス形式で検証する（非null検証では不十分）。**
  `assertNotNull($url)` は `"dummy"` のような無効値でも通過する。
  URL フィールドは `assertStringContainsString('/storage/customers/1/photo.jpg', $url)` のように**期待するパスを含むこと**まで確認する。
  - 期待結果の書き方: 「`photo_url` は `/storage/customers/{id}/photo.jpg` を含む文字列であること」（"存在する/non-null" で済ませない）

- **ルール3: すべての正常系 API テストでフィールド完全一致チェックを必須とする。**
  フィールド完全一致チェックを省略すると、内部フィールド（`photo_path`, `updated_at` など）の情報漏洩をテストが見逃す。
  `assertEqualsCanonicalizing(['field1', 'field2'], array_keys($body))` を**全正常系テストで**実施する（一覧APIの各要素にも適用）。
  - これは既存の「全フィールド一致で書く」ルールを**正常系で必須化**するもの。設計時点で各正常系ケースの期待フィールド集合を明記する。

### 暗黙の期待を排除する
テストケースの期待結果は、曖昧さのないように書く。
- 「正しいレスポンスが返る」→ NG（何が正しいのか不明）
- 「ステータスコード201、bodyに{id: integer, title: "入力値と同一"}が返る」→ OK

### 副作用を検証する（ステータスコードだけで成功と判定しない）
「成功レスポンスが返ること」だけを確認するテストは、**偽の成功（嘘をつくコード）を見逃す**。
副作用を伴う機能は、その副作用が**実際に起きたこと**まで検証するケースを必ず設計する。
- 登録/作成 → レスポンスだけでなく、**DBに利用可能な状態のレコードが存在し、後続機能（例: ログイン）が成功する**ことまで確認
- メール送信等の外部作用 → 送信処理が呼ばれたこと（モック検証）を確認。未実装なら「未実装である」ことをテストで固定する
- 認証エラー（401）→ 握り潰されず、**ログアウト等のリカバリが起きる**ことを確認
- 悪い例: 「パスワードリセット要求で200が返る」だけ → メール未送信でもPASSしてしまう
- 良い例: 「200が返り、かつメール送信処理が1回呼ばれる」／未実装なら「501が返る」

**fake/mock のイディオムは言語/FWで異なる。** アクティブなスタックプロファイル
（`.orchestrator/stack-profile.md`）の「テストダブル」節に従う。
- 例: Laravel `Mail::fake()` → `Mail::assertSent(...)` / `Queue::fake()` → `Queue::assertPushed(...)`
- 例: Python `unittest.mock.patch(...)` → `assert_called_once()` / 外部HTTPは `respx`
- 種別が**非同期ジョブ・イベント・CLI**の機能は、HTTPステータスではなく
  「ジョブが投入された／イベントが発火した／コマンドが副作用を起こした」ことを検証する。
