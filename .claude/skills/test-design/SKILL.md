# Skill: test-design（テスト仕様書生成）

## いつ使うか
- 設計書に基づいてテスト仕様書を生成するとき
- Gate 2で差し戻されてテスト仕様書を修正するとき

## 入力
- `docs/design/F-XXX.md` の設計書（実装コードは見ない）
- Tier情報（バックログから）

## 出力ファイル
`docs/test_spec/F-XXX.md` に以下の構造で保存する。

```markdown
# テスト仕様書: F-XXX（機能名）

## Tier: B
## テスト方針: 正常系・主要異常系・境界値・主要インフラエラーをカバー

## テストケース

### 正常系
| ID | テスト名 | 事前条件 | 入力 | 期待結果 | 優先度 |
|----|---------|---------|------|---------|--------|
| TC-F001-001 | ... | ... | ... | ... | 高 |

### 異常系
| ID | テスト名 | 事前条件 | 入力 | 期待結果 | 優先度 |
|----|---------|---------|------|---------|--------|
| TC-F001-002 | ... | ... | ... | ... | 高 |

### 境界値
（同上のテーブル形式）

### セキュリティ
（Tier S/A の場合はさらに厚くする）

### 時系列・並行性
（下記「時系列・並行性の観点表」から該当するものを必ず検討する）

### インフラ起因エラー
（外部サービス障害、DB障害等）

## 受入条件カバレッジ
| 受入条件 | カバーするテストケース |
|---------|---------------------|
| "条件1" | TC-F001-001 |
| "条件2" | TC-F001-002, TC-F001-003 |

## エラーカテゴリチェック
- [x] バリデーションエラー: TC-F001-00x
- [x] ビジネスロジックエラー: TC-F001-00x
- [ ] 認証・認可エラー: 対象外（理由: ...）
- [x] 外部サービス障害: TC-F001-00x
- [x] インフラ障害: TC-F001-00x
- [ ] タイムアウト: Tier B のため省略

## 時系列・並行性チェック
- [x] 同時実行・二度押し: TC-FXXX-00x
- [x] イベントの遅延・順序逆転・重複: TC-FXXX-00x
- [ ] 期限切れ資格情報での実行: 対象外（理由: ...）
```

## 重要なルール
- このSkillはテスト仕様書（何をテストするか）のみ出力する
- テストコード（どうテストするか）は implement Skill が生成する
- 実装コードは参照しない（テストファースト思想）
- 全受入条件がいずれかのテストケースでカバーされていること
- エラーカテゴリチェックリストを必ず含めること

## テスト設計の品質ルール

### 共通テスト品質ルールを必ず適用する（真実源を参照）
テストケースを設計する前に **`.claude/skills/_shared/test-quality-rules.md` を読み**、その内容を期待結果に反映する。
横断ルール（偽陽性アサーション禁止 ルール1〜3 / 全フィールド一致 / 副作用検証）はそのファイルが正。要点のみ再掲する:
- **全フィールド一致**: 期待結果に**全フィールド名を明記**し完全一致で書く（「idが含まれる」等の部分一致はNG）。
- **偽陽性アサーション禁止**: ①`assertNull`はキー存在確認（`assertArrayHasKey`）と分離 ②URLはパス形式で検証（`assertNotNull`不可） ③全正常系でフィールド完全一致チェック必須。
- **副作用検証**: ステータスコードだけで成功判定しない。副作用（メール送信・DB保存・状態変更）が実際に起きたことまで検証するケースを設計する。
- イディオム（アサーション/テストダブル）はスタックプロファイル（`.orchestrator/stack-profile.md`）の書式に読み替える。
- ルールの追加・学習記録の追記は `_shared/test-quality-rules.md` 側で行う（各SKILLには転記しない）。

### 時系列・並行性の観点表（設計書に書かれていなくても必ず検討する）
実装者とテスト設計者が同じ設計書から出発すると、設計書にないシナリオは両方から同時に抜ける。
以下は設計書由来でない**強制観点**として、該当する機能では必ずテストケース化を検討する
（該当しない場合は「対象外（理由）」を明記）。

| 観点 | 問い | 対象になりやすい機能 |
|------|------|--------------------|
| 同時実行 | 同じ操作が同時に2回来たら？（二度押し、同時POST） | 状態遷移、課金、パイプライン起動 |
| イベント遅延・逆転 | 通知/Webhookが操作より後に届いたら？先に別経路で状態が進んでいたら？ | Webhook、非同期ジョブ完了通知 |
| イベント重複 | 同一イベントが2回届いたら？（冪等性） | Webhook、キュー、リトライ |
| 期限切れ資格情報 | トークン/セッションが失効した状態でこのフローを通ると？ | 認証必須ページ、メールリンク、決済リダイレクト戻り |
| 再実行・再生成 | 一度成功した処理をもう一度実行したら？（課金は？キャッシュは？） | 再生成、再送、リトライ |
| 中断・部分失敗 | 複数ステップの途中で失敗したら、どの状態で止まりユーザーに何が見える？ | パイプライン、決済+DB更新 |

課金を伴う機能は「成功レスポンスが返る」だけでなく「**残高/回数が実際に減ること**」
「**2回目の実行でも減ること（または冪等で減らないこと）**」までテストケース化する。

### 暗黙の期待を排除する
テストケースの期待結果は、曖昧さのないように書く。
- 「正しいレスポンスが返る」→ NG（何が正しいのか不明）
- 「ステータスコード201、bodyに{id: integer, title: "入力値と同一"}が返る」→ OK
