# Skill: requirements（要件定義）

## いつ使うか
- 新しい案件の要件を整理するとき
- 顧客との会話メモから要件を構造化するとき
- 既存の要件に追加・変更があるとき

## 重要な前提
**このフェーズはAIだけでは完結しない。**
ユーザー（Kojiro）が顧客から聞いた情報を入力し、AIが構造化・抜け漏れチェック・確認質問を返す対話型のフェーズ。
AIが勝手に要件を推測して補完してはいけない。「わからないことはわからない」と明示し、ユーザーに確認する。

## フェーズの流れ

### Step 1: 生情報の受け取り

ユーザーから以下のいずれかの形式で情報を受け取る。どの形式でもOK。

- 顧客との会話メモ（テキスト貼り付け）
- 口頭で聞いた内容のざっくりした説明
- 既存の仕様書・提案書
- 参考にしたい既存サービスのURL
- 「こういうの作りたい」という一言

### Step 2: ヒアリングシートの生成

受け取った情報を分析し、**不足している情報を質問としてまとめる**。
以下のカテゴリで抜け漏れをチェックする。

```markdown
# ヒアリングシート: （案件名）

## 1. ビジネス文脈（なぜ作るのか）
- [ ] 誰が使うのか（エンドユーザーの属性）
- [ ] 何を解決するのか（現状の課題）
- [ ] 成功の基準は何か（KPI・ゴール）
→ 不明な点:（ここにAIが質問を書く）

## 2. 機能要件（何を作るのか）
- [ ] 主要な機能の一覧
- [ ] 各機能の入力と出力
- [ ] 優先度（なくてはならない / あると良い / 将来的に）
→ 不明な点:

## 3. 非機能要件（どう作るのか）
- [ ] 技術スタックの指定 or 制約
- [ ] パフォーマンス要件（同時アクセス数、レスポンス時間）
- [ ] セキュリティ要件（認証、個人情報の有無）
- [ ] 可用性要件（ダウンタイム許容度）
→ 不明な点:

## 4. 制約条件
- [ ] 予算
- [ ] 納期
- [ ] 既存システムとの連携
- [ ] 法規制・コンプライアンス
→ 不明な点:

## 5. 運用・保守
- [ ] 誰がデプロイするか
- [ ] 監視・ログの要件
- [ ] データバックアップ
→ 不明な点:
```

**全項目を埋める必要はない。** 案件の規模に応じて必要な項目だけ確認する。
小規模（light）なら 1, 2 が最低限。中規模（medium）以上なら 3, 4, 5 も確認する。

### Step 3: 要件の構造化

ユーザーとの対話で情報が揃ったら、以下のファイルを生成する。

#### docs/requirements.md

```markdown
# 要件定義書: （案件名）

## 1. 案件概要
- 案件名: 
- 顧客: 
- 目的: （何を解決するか。1〜2文で）
- エンドユーザー: 
- 成功基準: 

## 2. 技術スタック
- 言語: 
- フレームワーク: 
- DB: 
- フロントエンド: 
- インフラ: Docker
- 理由:（なぜこの構成を選んだか。顧客指定 or 提案）

## 3. 機能一覧
| ID | 機能名 | 優先度 | Tier | 概要 | 受入条件 |
|----|--------|--------|------|------|---------|
| F-001 | xxx | 高 | B | ... | 1. xxx 2. xxx 3. xxx |

## 4. 非機能要件
（該当するもののみ記載）

## 5. 制約条件
（該当するもののみ記載）

## 6. 未決事項
| # | 内容 | 影響範囲 | 確認先 | ステータス |
|---|------|---------|--------|-----------|
| 1 | xxx | F-002に影響 | 顧客に確認 | 未確認 |

## 7. 前提・除外事項
- 前提: ...
- スコープ外: ...（明示的に「やらないこと」を書く）
```

### Step 4: 要件のセルフチェック

要件定義書を生成した後、AIが以下の観点でセルフチェックを行い、結果をユーザーに報告する。

| チェック項目 | 内容 |
|------------|------|
| 受入条件の具体性 | 全機能の受入条件が「テスト可能な形」で書かれているか。「使いやすい」はNG、「3クリック以内で操作完了」はOK |
| 機能間の依存関係 | 機能間の依存が明示されているか。暗黙の依存がないか |
| Tier判定の妥当性 | 金額・個人情報・認証に関わる機能が Tier S/A に設定されているか |
| 未決事項の影響 | 未決事項が残ったまま設計に進んでも大丈夫か。設計を止めるべき未決事項がないか |
| スコープの明確さ | 「やること」と「やらないこと」の境界が明確か |

### Step 5: ユーザー承認

セルフチェック結果を報告し、ユーザーに最終確認を求める。

```
要件定義書を作成しました。セルフチェック結果:
- 受入条件の具体性: ✅ OK
- 機能間の依存関係: ✅ OK
- Tier判定: ⚠️ F-003の決済機能が Tier B になっています。S にすべきでは？
- 未決事項: ⚠️ 1件あり（メール送信のSMTP設定）。設計には進めますが、実装前に確定が必要です。
- スコープ: ✅ OK

要件定義書の内容を確認してください。
OKであれば「承認」と指示してもらえれば、設計フェーズに進みます。
修正があれば指示してください。
```

**ユーザーが「承認」するまで設計フェーズに進まない。**

## AIがやってはいけないこと
- 顧客の業務ロジックを推測で補完する（「一般的にはこうだから」で埋めない）
- 未確認の要件を「たぶんこうだろう」で進める
- ユーザーが理解していない要件をそのまま設計に流す
- 受入条件を曖昧なまま通す

## AIが積極的にやるべきこと
- 「この情報が足りないので顧客に確認してください」と具体的に伝える
- 「この要件は矛盾しています」と指摘する
- 「一般的にはこういう要件も考慮しますが、この案件では必要ですか？」と提案する
- 技術スタックの選定理由を説明し、ユーザーが顧客に説明できるようにする
- 「これはスコープ外にした方が納期に間に合います」とトレードオフを提示する
