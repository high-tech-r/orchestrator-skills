各SKILL.mdには要点のみ記載し、詳細・根拠はここを正とする。ルールの追加・変更はこのファイルで行う。

## A. 偽陽性を生むアサーションの禁止（F-031 自己レビューで発見）
「PASSしているのに実は何も保証していない」アサーションを設計・生成しない。

### ルール1: null チェックはキー存在確認と分離する
`$res->json('key')` はキー不在でも `null` を返すため、`assertNull` だけでは
「キーが存在して null」と「キーが存在しない」を区別できない。
必ず `assertArrayHasKey('key', $body)` でキー存在を先に確認してから `assertNull($body['key'])` を行う。

### ルール2: URL フィールドは `assertNotNull` ではなくパス形式で検証する
`assertNotNull($url)` は `"dummy"` のような無効値でも通過する。
URL フィールドは `assertStringContainsString('/storage/customers/1/photo.jpg', $url)` のように
**期待するパスを含むこと**まで確認する（"存在する/non-null" で済ませない）。

### ルール3: すべての正常系 API テストでフィールド完全一致チェックを必須とする
フィールド完全一致を省略すると、内部フィールド（`photo_path`, `updated_at` など）の情報漏洩を見逃す。
`assertEqualsCanonicalizing(['field1', 'field2'], array_keys($body))` を**全正常系テストで**実施する
（一覧APIは各要素にも適用）。これは下記Bルールの正常系での必須化であり、省略不可。

## B. レスポンス検証は「全フィールド一致」で書く
APIレスポンスの検証は、含まれるべき**全フィールド名を明記**して完全一致で書く。
`"field" in body`（部分一致）は禁止。フィールドの追加・削除・名前変更を検出できないため。
- 悪い例: 「レスポンスにidとtitleが含まれている」→ 余計なフィールドが増えても気づかない
- 良い例: 「レスポンスのフィールドは {id, title, description, is_completed, created_at} と完全一致する」
- コード: `set(body.keys()) == {全フィールド}` / `assertEqualsCanonicalizing([...], array_keys($body))`

## C. 副作用を検証する（ステータスコードだけで成功と判定しない）
「成功レスポンスが返ること」だけを確認するテストは、**偽の成功（嘘をつくコード）を見逃す**。
副作用を伴う機能は、その副作用が**実際に起きたこと**まで検証する。
- 登録/作成 → レスポンスだけでなく、**DBに利用可能な状態のレコードが存在し、後続機能（例: ログイン）が成功する**ことまで確認
- メール送信等の外部作用 → 送信処理が呼ばれたこと（モック検証）を確認。未実装なら「未実装である」ことをテストで固定する
- 認証エラー（401）→ 握り潰されず、**ログアウト等のリカバリが起きる**ことを確認
- 悪い例: 「パスワードリセット要求で200が返る」だけ → メール未送信でもPASSしてしまう
- 良い例: 「200が返り、かつメール送信処理が1回呼ばれる」／未実装なら「501が返る」

**fake/mock のイディオムはスタックプロファイルの「テストダブル」節に従う。**
- 例: Laravel `Mail::fake()` → `Mail::assertSent(...)` / `Queue::fake()` → `Queue::assertPushed(...)`
- 例: Python `unittest.mock.patch(...)` → `assert_called_once()` / 外部HTTPは `respx`
- 非同期ジョブ・イベント・CLI の機能は、HTTPステータスではなく
  「ジョブが投入された／イベントが発火した／コマンドが副作用を起こした」ことを検証する。
