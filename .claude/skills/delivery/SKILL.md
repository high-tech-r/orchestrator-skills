# Skill: delivery（納品ドキュメント・パッケージ生成）

## いつ使うか
- ステークホルダーへの納品時に、渡すドキュメント一式を `docs/delivery/` にまとめるとき
- 「納品ドキュメントを作って」「納品パッケージを用意して」と指示されたとき

機能単位のパイプライン外。リリース・納品のタイミングで単発実行する。

## 役割
`docs/delivery/` に**ステークホルダーが読む成果物だけ**を集約する。開発者向けの設計書・整合性レポートは
`docs/` 配下のまま残し、こちらにはステークホルダー向けに要約・整形したものを置く。

## 生成・収集するもの

| ファイル | 生成方法 |
|---|---|
| `docs/delivery/index.md` | 本スキルが生成（納品物一覧） |
| `docs/delivery/design_summary.md` | 本スキルが生成（ステークホルダー向け設計サマリー） |
| `docs/delivery/quality_report_YYYY-MM-DD.md` | `quality-report` スキルで生成（無ければ呼ぶ） |
| `docs/delivery/security_report_YYYY-MM-DD.md` | `security-report` スキルで生成（無ければ呼ぶ） |
| `docs/delivery/presentation_YYYY-MM-DD.md` | 本スキルが生成（**Gamma取り込み用スライドMarkdown**） |
| （任意）共有Webページ | 本スキルが生成（claude.ai の **Artifact**。共有リンクでステークホルダーに渡す。**外部公開・opt-in**） |

## 手順
0. **納品前チェック（推奨）**: 直近のリリース内容に対して `audit` スキル（敵対的コードベース監査）が
   未実行なら、実行して blocking（高深刻度）指摘ゼロを確認するようユーザーに提案する。
   未対応の高深刻度指摘が残る場合は、納品前にユーザーへ明示する。
1. `docs/delivery/` が無ければ作成する。
2. `quality_report` / `security_report` が未生成なら、それぞれのスキルを実行して生成する。
3. `design_summary.md` を生成する（下記）。
4. `presentation_YYYY-MM-DD.md` を生成する（Gamma取り込み用。下記）。
5. `index.md` を生成する（下記）。
6. 生成した納品物の一覧をユーザーに提示する。
7. （任意）ユーザーが**共有リンク**を望む場合のみ、集約レポートを Artifact として公開する
   （下記「共有Webページ（Artifact）」）。求められなければ実行しない。

### design_summary.md（ステークホルダー向け設計サマリー）
`docs/design/F-XXX.md`（開発者向け）をステークホルダー向けに要約する。技術詳細ではなく
「何ができるか」を中心に。

```markdown
# 設計サマリー
- 対象: （プロジェクト名）
- 作成日: YYYY-MM-DD

## システム概要
（何のためのシステムか。1〜2段落）

## 提供機能一覧
| 機能 | 概要（ステークホルダーの言葉で） | 主な入出力 |
|------|--------------------|-----------|
| F-001 タスク追加 | タイトルと説明でタスクを登録できる | 入力: タイトル等 / 出力: 登録結果 |

## 技術構成（参考）
- 言語/FW/DB/インフラ（Docker）を簡潔に
```

### presentation_YYYY-MM-DD.md（Gamma取り込み用スライド）
ステークホルダー説明用のスライド原稿。**Gamma（gamma.app）にMarkdownとして取り込むことを前提**にする。

**Gamma取り込みの約束ごと（厳守）:**
- **`---`（水平線）が1スライドの区切り。** スライドごとに `---` を必ず挟む。
- 各スライド冒頭の `#`/`##` 見出しがスライドのタイトルになる。1スライド1見出し。
- 中身は**箇条書き中心・1行を短く**（1スライドに詰め込みすぎない。目安5〜7項目）。
- 画像/表の多用は避け、テキストで完結させる（Gamma側でレイアウトされる）。
- 数値・合否は品質/セキュリティレポートと一致させ、不利な事実も載せる（誠実性）。

```markdown
# （プロジェクト名）納品報告
（提出先名） / 納品日 YYYY-MM-DD

---

## 本日お伝えすること
- 提供する機能
- 品質の担保（テスト結果）
- セキュリティの担保（検査結果）
- 残課題と今後

---

## システム概要
- 目的: （1行）
- 構成: 言語 / FW / DB（Docker）

---

## 提供機能
- F-001 タスク追加: （ステークホルダーの言葉で1行）
- F-002 タスク一覧: （1行）

---

## 品質の担保
- テスト: PASS n/n（カバレッジ x% ／未計測なら「未計測」と明記）
- 全機能が品質ゲート（Gate 1/2）を通過
- 既知の不具合: なし ／ n件（重大度）

---

## セキュリティの担保
- 秘密情報・脆弱性・ライセンス・静的解析を自動検査し品質ゲート通過
- SBOM・ライセンスレポートを証跡として添付（OWASP Top10 / CWE 対応）

---

## 残課題・今後
- （許容したリスク・未対応・次フェーズ。正直に記載）

---

## ご確認のお願い
- 受け入れテスト手順書: docs/review_map/
- 納品物一覧: docs/delivery/index.md
```

### index.md（納品物一覧）
```markdown
# 納品ドキュメント一覧
- 対象: （プロジェクト名 / リリース or コミットSHA）
- 納品日: YYYY-MM-DD

| ドキュメント | 内容 | ファイル |
|------------|------|---------|
| 設計サマリー | 提供機能と構成 | design_summary.md |
| 品質レポート | テスト・バグ・ゲート通過状況 | quality_report_YYYY-MM-DD.md |
| セキュリティレポート | 検査内容と証跡 | security_report_YYYY-MM-DD.md |
| 説明スライド | ステークホルダー説明用（Gamma取り込み用Markdown） | presentation_YYYY-MM-DD.md |
| SBOM / ライセンス | 部品表・ライセンス一覧 | （CI成果物 sca-evidence を別添） |

## 本システムの品質・セキュリティの担保方法（要約）
（SECURITY.md と品質レポートの要点を3〜5行で。ステークホルダーが最初に読む前提）
```

### 共有Webページ（Artifact）※任意・opt-in・外部公開
非技術のステークホルダーに**共有リンク**で見せたい場合、集約レポートを claude.ai の
**Artifact**（既定プライベートなWebページ）として公開できる。閲覧者はブラウザで開けるため、
リポジトリの Markdown を渡すより伝わりやすい。**Markdown 納品物を置き換えるものではなく、その上の
表示層**（加算レイヤー）。

**必ず守る（守秘・opt-in・loud）:**
- **既定では実行しない。** ユーザーが明示的に「共有リンクにして」と求めたときだけ公開する（手順7）。
- 公開は **claude.ai（外部サービス）への送信**。既定プライベートだが共有可能で、キャッシュ・
  インデックスされ得る（後から消しても残り得る）。**顧客の納品物を外部へ出す判断はユーザーが行う。**
  何を載せるか（機能一覧・品質/セキュリティの要点）を事前に提示し、**機密（内部URL・認証情報・
  個人情報・接続文字列等）は載せない**。
- 元データは `docs/delivery/` の Markdown を正とする。**Artifact の数値・合否はそれと一致**させ、
  不利な事実も削らない（二重帳簿禁止＝誠実性）。

**作り方:**
1. 先に `artifact-design` スキルを読み、投資度（作り込み量）を案件に合わせる。
2. 集約内容（`index` ＋ `design_summary` ＋ 品質/セキュリティの要点）を**1枚のHTML**にまとめる。
   自己完結（外部CDN・フォント・画像を使わない）／テーマ対応（light・dark両対応）／`<title>` は安定。
3. Artifact ツールで公開し、**共有リンクをユーザーに渡す**（誰に共有するかはユーザーが決める）。

**フォールバック（loud）:** Artifact が使えない環境（ヘッドレス・ツール無効等）では共有ページは
作らず、**その旨を明示する**。Markdown 納品物は常に生成済みなので**納品自体は成立する**。

## 重要なルール（誠実性）
- 納品物に書く合否・件数は、品質/セキュリティレポートの実データと一致させる。**二重帳簿を作らない。**
- 共有Webページ（Artifact）は claude.ai への**外部公開**。ユーザーの明示指示があるときだけ行い、
  機密は載せず、数値は `docs/delivery/` の Markdown と一致させる（同上の二重帳簿禁止）。
- 未実施・未計測・残課題は index でも隠さない（「品質レポート参照」で逃げず、要点は index にも出す）。
- ステークホルダー向けに平易化はするが、**不利な事実を消す平易化はしない。**
- Gamma等でスライド化する場合も、元データは本フォルダのMarkdownを正とする（現時点では手動エクスポート）。
