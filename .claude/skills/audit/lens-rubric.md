# 観点ラブリック L1〜L9（言語非依存の網羅監査チェックリスト）

`audit` スキルの**核**。各サブエージェント（または各監査セッション）に**この全文を明示的に**渡す
（「バグを探して」ではなく「このリストで各対象を1つずつ検証せよ」）。骨子は普遍。
スタック固有の「例」だけアクティブなスタックプロファイル（`.orchestrator/stack-profile.md`）に応じて差し替える。

> 使い方: 各レンズは「**列挙 → 1件ずつ判定**」の2段階で実行する。列挙を省略して「目についたもの」
> だけを見ると弱いモデルほど網羅が崩れるため、**先に列挙結果（grep 出力・一覧表）を書き出してから**
> 判定に入ること。「該当なし」と結論する場合も列挙結果を示す。

---

## L1. 認可・入力境界（マルチテナント IDOR — 最頻出・最重要）
列挙するもの: **全ての書き込み系エンドポイント**と、ID でリソースを取得する全箇所。
- ルーティングのモデル束縛 / ORM の ID 取得で受けたリソースの**所有者を検証しているか**。
  **全 CRUD 動詞を1つずつ確認**（index/show/create/store/edit/update/destroy）。
  片方だけ守られている「**sibling 取りこぼし**」を特に疑う（destroy は守るが update は素通し 等）。
- 「他モデルを PK で参照するフィールド」（例: 住所 ID、リソース ID）の queryset/クエリが
  ユーザーでスコープされているか（IDOR）。一覧・検索・集計・エクスポートも対象。
- 一意制約（unique）が**テナント跨ぎ**になっていないか（他テナントの存在推測＝列挙、正規データ登録の阻害）。
- 公開（未認証）ルートが非公開データを露出していないか。連番 ID・推測可能なファイル名での列挙。
- 検証ゲートのバイパス経路: 属性を変更する API が検証フラグをリセットしているか
  （例: メール変更後も verified のまま）。create 時と update 時でバリデーションが同じか。
- **スタック例**:
  - *Laravel*: `Model $x` バインドに所有権チェック（`$x->user_id === auth()->id()` / `abort_unless` / Policy）。`unique:t,c` を `Rule::unique()->where('user_id',...)` に。ルートの `middleware(['auth'])` 付け忘れ。
  - *Rails*: `before_action` の認可、`current_user.things.find(params[:id])`（スコープ経由取得）か素の `Thing.find` か。Strong Parameters。
  - *Django/DRF*: QuerySet のテナントフィルタ、`get_object_or_404(qs_scoped, ...)`、Permission クラス。
  - *クライアント(SPA/モバイル)*: サーバ認可を前提にしつつ、画面から他ユーザー/他テナント ID を送れる導線がないか。

## L2. 実装の誠実性・失敗パス（偽成功・握り潰し）
列挙するもの: 対象領域の**全ての** `catch` / `except` / `rescue` / `.catch()` / フォールバック /
空エラーハンドラ / `fail_silently` / デフォルト値による代替。
判定: 1件ずつ「**この行が失敗したらユーザーに何が見えるか**」に答える。
- 失敗がログにも呼び出し元にも伝わらない → 握り潰し（blocking 相当）
- フォールバックが「成功」として報告される → **偽成功（最重大）**。例: 処理失敗時に元データを
  そのまま返して completed と報告、エラーページの HTML を成功した画像として返す
- 未実装/失敗なのに「成功」を返す。空実装のまま生きているルート・ハンドラ。
  成功メッセージ（「送信しました」）が**実際に処理完了した場合のみ**出るか
- 失敗を成功ステータスで**永続化**していないか（生成失敗を `completed` 保存、空結果を成功表示）
- **認証エラー（401）を握り潰してリカバリしない**（ログアウト→ログイン導線に落とすべき）
- 外部から取得した値を検証せず使う（`raise_for_status` 無し、Content-Type 未確認）→ 偽成功の温床
- 壊れた状態のデータ作成（後続機能が動作しない前提のレコードを「作成成功」にする）

## L3. 境界契約の突合（cross-boundary contract check）
単体では正しいコード同士の組み合わせで壊れるバグを探す。**境界を列挙し、両側を同時に開いて突合する。**
列挙するもの:
- サーバー設定 ⇔ クライアント実装の対（例: トークンローテーション設定 ⇔ トークン保存処理、
  CORS 設定、内部 URL の変換処理）
- サービス間 API（レスポンス形状 ⇔ クライアント側モデル/パーサ、タイムアウト、ジョブポーリングの契約）
- 同一リソースを更新する複数経路（例: 管理画面操作 ⇔ Webhook ⇔ ユーザー API が同じステータスを触る）
- 環境差（コンテナ内部ホスト名、開発/本番 URL、ビルド時定数）
判定: 対の両側を読み、「片側の前提をもう片側が満たしているか」を1対ずつ確認する。
**片側だけ読んで「たぶん合っている」は禁止。**

## L4. 並行性・時系列
「操作が重なる・遅れる・二重に届く」ケースを機械的に列挙する。
- **check-then-act**: 状態を確認してから変更するコードで、確認と変更が同一ロック/
  トランザクション内に無いもの（同時 POST・二度押しで二重実行。カウンタ・在庫・満席判定の競合、TOCTOU）
- **イベント到着順の逆転・遅延・重複**: Webhook/非同期通知のハンドラが「現在状態」を
  確認せずに状態を上書きしていないか（例: 遅延 Webhook が後段ステータスを巻き戻す）。
  ハンドラは冪等か（同一イベント2回で安全か）
- **資格情報の期限切れ**: 期限切れトークン/セッションが残った状態で各フロー
  （公開ページ、リセットリンク、決済リダイレクト戻り）を通るとどうなるか
- **多重リフレッシュ/多重送信**: 同時 401 でのトークンリフレッシュ、ポーリングの
  停止条件（エラー時に止まるか、画面離脱で止まるか）

## L5. 不在の検出（absence check）
「あるべきものが無い」は順読みでは見えない。**grep による否定の証明**を行う。
列挙するもの（それぞれ「定義箇所」と「使用箇所」を grep で突合）:
- 宣言・呼び出しされているが**どこからも代入されていない**コールバック/フック
- モデル/テーブルはあるが**ビュー/API が無い**（書き込み専用データ = 機能しているふり）
- ステータス値・enum 値はあるが**そこへ遷移させるコードが無い**
- 受け取るが**何にも影響しないパラメータ**（嘘のインターフェース）
- トランザクション欠落・順序の誤り（先に保存→後で検証）

## L6. リソースのライフサイクル
- 無限成長するもの: in-memory の辞書/リスト、ログ、一時ファイル、ジョブの入出力ファイル
  （「削除 API がある」だけでは不十分 — **誰がいつ呼ぶのか**まで確認する）
- キャッシュの無効化: キャッシュキーに「変わり得る変数」が全て含まれているか
  （例: URL が変わったのに ID だけをキーにして古い内容を出し続ける）
- CPU 重処理がイベントループ/メインスレッドを塞いでいないか（処理中にヘルスチェックが応答できるか）

## L7. フレームワーク/ライブラリの誤用
- SPA ↔ 非 SPA ページ間の遷移方法の誤り（例: *Inertia* で非 Inertia ページへ `redirect()`/`<Link>`
  → モーダル化・リロード必須。正しくは `Inertia::location()` かフルページ遷移。Next.js の
  `router.push` vs `<a>`、等）
- 設定/環境値の読み方（例: 本番 config キャッシュ後に **`env()` 直読み → null**。`config()` 経由にする）
- ルート/ハンドラの**重複定義**（同一 method+path で後勝ち。認証あり版が死んで認証なし版が動く等）
- ORM のキャスト/型不整合（TIME 列を datetime キャスト等）、シリアライズのキー名不一致

## L8. 正しさ・壊れフロー・パフォーマンス
- 存在しないルート/カラム/メソッドへの遷移・呼び出し（実行時 500）
- 削除済み/未取得モデルへの操作（無言 no-op）、mass-assign / 一括代入の過剰
- N+1（ループ内クエリ）
- 日時/タイムゾーン/月末/重複の境界。**破壊的な日時オブジェクト操作**（`addHours` を copy せず二重適用 等）
- **クライアント**: null 安全、未マウントでの状態更新、無限ローディング、サーバ確定前の楽観 UI、入力の消失

## L9. デッドコード・残骸・重複（リファクタ）
- 未使用の関数/クラス/ルート/コンポーネント/スキャフォールド（**未使用は無害ではなく攻撃面**。
  プロジェクトルールに反する死蔵コードは即報告 — 誤配線されると事故になる）
- デバッグ残骸（`dd()`/`var_dump`/`console.log`/`print`）、**PII を吐くログ**（リクエスト全体のログ等）
- コピペ増殖（同一ロジックの多重複 → 挙動差の温床）。巨大関数/巨大コンポーネント
- ベーステンプレート由来の未使用機能（雛形のセキュリティ穴ごと残っていないか）

---

## なぜこの構成なのか
2つの導入プロジェクト（2026-07）で実際に検出された欠陥を一般化した結果。
**L1（IDOR）と L2（嘘の成功）が件数・深刻度とも支配的**だった。L1 は「全動詞・全ルートを機械的に潰す」
ことでしか網羅できず、L2 は「成功表示は実処理完了時のみ」という単純規則の違反として機械的に拾える。
L3〜L4 は「単体では正しいコードの組み合わせ・タイミングで壊れる」バグ（機能単位 Gate の構造的死角）、
L5〜L9 が残りの実害・保守性を掬う。このリストを**固定で毎回渡す**ことが、モデル間のばらつきを消す
最大のレバーになる。
