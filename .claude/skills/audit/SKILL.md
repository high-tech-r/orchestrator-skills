# Skill: audit（敵対的コードベース監査 — 全域バグ・リファクタ網羅）

## いつ使うか
- 機能開発が一巡したとき（複数機能がマージされた後。目安: 大きめの機能追加 3〜5 件ごと）
- リリース前チェックリストの実行時
- 定期実行（スプリント末 / 月次 / 四半期のいずれか早い方）
- ユーザーが「バグを一斉に探して」「監査して」「リファクタ対象を洗い出して」と依頼したとき
- 過去のレビューやテストの**取りこぼし**（sibling メソッド・別コントローラの同型欠陥）を潰したいとき

**機能単位の Gate 1/2 とは別物。** Gate は「設計どおりに作られたか」の答え合わせであり、
本スキルは「バグは必ずある」前提の**反証探索**である。Gate を全て PASS したコードベースからも
バグは見つかる（導入プロジェクトの実績: Gate/テスト全通過のコードから 70 件超・重大 18 件を検出）。

### 位置づけ（二層防御）
| | 対象 | 見方 | いつ |
|---|---|---|---|
| `consistency-check`（Gate 2） | 実装中の**1機能** | 設計⇔コード⇔テストの整合（縦・狭深） | 各機能の実装+テスト後 |
| **`audit`（本スキル）** | **全域横断** | 境界間・並行性・IDOR・偽成功・デッド/重複（横・広網） | 大型機能後・リリース前・定期 |

機能単位パイプラインの構造的死角を塞ぐ:
1. 機能単位レビューの死角は機能間・サービス間の「契約」（両側を同時に読む工程が存在しない）
2. 実装とテストが同じ設計書から出発するため、設計書にない敵対的シナリオ
   （イベント遅延・二度押し・期限切れトークン）は両方から同時に抜ける
3. 「バグは必ずある前提の反証探索」という工程がそもそも無い

## 前提思想 ── 検出力はモデルではなく「プロセス」で決まる
網羅監査の成果は、使う LLM の銘柄より **「観点を固定リストで明示したか」「並列で全読みしたか」
「敵対的に再検証したか」** で決まる。強いモデルでも「バグを探して」だけでは拾いが不安定。
逆に下記の型を踏めば、どのモデルでも安定して同水準を出せる。本スキルはその型を**強制**する。

## 基本原則
1. **反証探索として実行する。** 「問題ないことの確認」ではなく「壊れる条件の発見」を目的にする。
   指摘ゼロで終わる監査は、観点の消化不足を疑う。
2. **裏取りできた指摘のみ報告する。** 指摘には必ず「読んだ根拠コード（file:line）」と
   「具体的な失敗シナリオ（入力/状態 → 誤動作）」を付ける。推測・憶測は棄却する。
3. **修正はしない。** 監査はレポートのみ（発見と修正を分離）。修正はバックログ登録後、
   ユーザーの優先度判断を経て別ブランチで行う。
4. **観点は1つずつ・列挙してから判定する。** 全観点を一度に見ようとすると浅くなる。
   `lens-rubric.md` のレンズ L1〜L9 を1つずつ、「列挙 → 1件ずつ判定」の2段階で対象領域の
   全ファイルに適用する。

## 実行手順

### Step 1: スコープ確定・領域分割
- 監査対象を確認（全域か、特定領域か）。**直近で監査済みの領域は除外**して重複を避ける。
- コードベースをサブシステムに分割する。目安は「関心事＋レイヤ」で 3〜8 領域。
  各領域が**1エージェントで全読みできる粒度**にする。
  例: ① 認証・課金・管理（バックエンド）② ドメインロジック・ジョブ・メール（バックエンド）
  ③ 公開ルート＋ルート横断 ④ モバイル ⑤ Web フロントエンド
- **既知の設計前提**（誤検出回避用）を用意する（例: 「管理者判定はファイル許可リストで正しい」
  「ローカルキューは sync」「テナントは user_id 列」等）。

### Step 2: 並列サブエージェントで全読み監査（fan-out）
- サブシステムごとに**読み取り専用**の調査エージェント（Explore 等）を**並列**に起動する
  （並列が使えない環境では1領域ずつ逐次でも可。ただし**1セッション＝1領域**を守る）。
- 「locate（在り処探し）」ではなく「担当領域を**丸ごと読んで**ラブリックで**突合**せよ」と指示する。
- **専門エージェント（`.claude/agents/`）を活用する。** 高価値レンズは system prompt に観点を
  内蔵した名前付きエージェントを使うと、モデル間のばらつきが減り再現性が上がる:
  - `authz-idor-auditor` → L1（認可・IDOR。全動詞巡回・sibling 取りこぼし）
  - `honesty-auditor` → L2/L5（偽成功・握り潰し・嘘のインターフェース）
  - `contract-checker` → L3/L4（境界契約・並行性）
  使い方は2通り: (a) **横断パス** — コードベース全域に対しこの3体を並列起動し、支配的な
  L1/L2/L3/L4 を機械的に潰す。(b) **領域割り当て** — その領域がこれらの関心で支配的なら、
  subsystem リーダーとして該当エージェントを充てる。残りレンズ（L6〜L9）は汎用エージェントで補う。
- 各エージェントへのプロンプト雛形（`{{...}}` を埋める）:

```
{{プロジェクト名・スタック}} の「{{担当サブシステム名}}」を very thorough で監査し、バグ・不整合・
リファクタ候補を洗い出す。**コードは一切変更せず報告のみ**。{{除外領域}}。

## 担当ファイル
{{対象ファイル/ディレクトリの列挙}}

## 観点（この全項目で各対象を1つずつ検証。「探す」ではなく「潰す」）
{{lens-rubric.md の L1〜L9 全文を貼る}}

## 既知の設計前提（誤検出回避）
{{例: 管理者判定はファイル許可リストで正しい / ローカルキューは sync / テナントは user_id 列 …}}

## 出力様式
1. 冒頭に対象構成の要約（3〜5行）
2. 「バグ」と「リファクタ候補」を分けて列挙
3. 各項目: file:line ＋ 症状/根拠（1〜2文）＋ 深刻度（高/中/低）
4. 誤検出回避のため「問題なしと確認した点」も簡潔に記録
```

### Step 3: 統合・重複排除・深刻度ランク
- 全エージェントの結果をマージし、同一箇所の重複を排除する。
- **横断テーマ**を抽出する（例: 同型 IDOR が N 箇所、嘘の成功が M 箇所）。
- 深刻度でランク付け: 高（セキュリティ/データ整合/必ず壊れる）→ 中 → 低/リファクタ。

### Step 4: 敵対的再検証（雑音の除去）
- 高深刻度の各指摘に対し、別エージェント（または自分）で**反証を試みる**。
  指摘のレンズに対応する専門エージェント（L1→`authz-idor-auditor` / L2・L5→`honesty-auditor` /
  L3・L4→`contract-checker`）を**検証役**に充てると、観点が揃い反証の質が上がる:
  - そのルートは本当に到達可能か
  - 所有権チェックが別レイヤ（ミドルウェア/ポリシー）に無いか
  - その機能は使われているか（**未使用なら「修正」ではなく「撤去」が正**）
  - 再現手順は成立するか
- 根拠コードを再読して成立しない指摘は落とす。生き残った指摘のみ確定し、
  確度を CONFIRMED / PLAUSIBLE でタグ付けする。

### Step 5: レポート出力・バックログ登録（成果の固定）
- `docs/audit/YYYY-MM-DD_<領域>.md` に領域別レポート、`docs/audit/YYYY-MM-DD_summary.md` に統合サマリー。
- 確定した指摘を `docs/backlog.md` に **TD（技術的負債）項目として採番登録**し、
  `.orchestrator/project_status.yaml` にも反映する。各項目に必ず:
  **`file:line`**（着手時にそのまま作業できる粒度）・深刻度・**対処方針**（修正 / 未使用なら撤去 /
  リファクタ）・バグ↔リファクタの区別。
- 修正はユーザーが優先度を判断してから（監査スキル自身は修正しない）。

### Step 6（推奨）: 第2ラウンド以降 — NET-NEW の機械巡回
余裕があれば **loop-until-dry**: 新規発見が途絶えるまでレイヤ/観点を変えて追加監査する。
第2ラウンド以降はプロンプトに次を追加する（実証済みで効果大）:

```
## これは第2ラウンド。目的は NET-NEW（未登録の取りこぼし）の確定。
- 既知の以下は**再掲しない**。これら以外を探せ:
  {{第1ラウンドで登録済みの指摘リスト（ID＋一言）}}
- **全CRUD動詞・全ルート・全{{画面/モデル/Request/Migration}}を機械的に1つずつ巡回**し、
  片方だけ守られている sibling 取りこぼしを特に疑う。
- 前回コントローラ中心だった場合は、**Model / FormRequest / Migration / Middleware 層**を別建てで巡回する。
- 冒頭に「確認した対象数／うち安全と確認・問題ありの内訳」を出し、カバレッジの証跡を残す。
```

導入プロジェクトの実績: 良質な第1ラウンドの後でも、第2ラウンドの機械巡回が
高深刻度1件（課金導線が実質機能していない）・sibling 取りこぼし・未監査レイヤの欠陥を追加検出した。

## 指摘のフォーマット（必須項目）

```markdown
### [重大度] 指摘タイトル
- 場所: path/to/file.py:123
- 類型: C1〜C10（下表）
- 検出レンズ: L1〜L9
- 欠陥: （1文）
- 失敗シナリオ: 入力/状態 → 何が起きるか → ユーザー/事業への影響
- 根拠: 読んだコードの要点（該当行の引用または要約）
- 確度: CONFIRMED / PLAUSIBLE
```

### バグ類型（共通スキーマ — プロジェクト横断で集計・比較に使う）
| 類型 | 名称 | 定義 |
|------|------|------|
| C1 | 偽成功 | 失敗・未実装を成功として報告する |
| C2 | 握り潰し | エラーがログにも呼び出し元にも伝わらない |
| C3 | 境界契約不整合 | 設定⇔クライアント、サービス間、複数更新経路の食い違い |
| C4 | 並行性・時系列 | race、イベント順序逆転・遅延・重複、二重実行 |
| C5 | 不在 | 宣言・定義だけあり配線/実装/参照が無い |
| C6 | 認可・セキュリティ | IDOR、検証バイパス、推測可能な公開リソース |
| C7 | リソースライフサイクル | リーク、無限成長、キャッシュ無効化漏れ、ループブロッキング |
| C8 | 嘘のインターフェース | no-op パラメータ、ルール違反の死蔵コード |
| C9 | 誤用・壊れフロー | フレームワーク/ライブラリの誤用、存在しない参照、型不整合 |
| C10 | 重複・肥大 | コピペ増殖、巨大関数、デバッグ残骸、未使用コード（リファクタ） |

統合サマリーには**類型×重大度のクロス集計表**と「従来プロセス（Gate/テスト）をすり抜けた理由」の
分類を必ず含める。この形式を揃えることで、複数プロジェクトの監査結果を横断集計できる。
カバレッジの副産物（「どこが安全と確認できたか」）も証跡付きで記録する —
**「穴」と「安全」を網羅的に言い切れる**状態が監査のゴール。

## 実行時の注意（モデル性能に依存しないための規律）
- **1セッション（または1サブエージェント）= 1領域。** コンテキストを跨がせない。
- **レンズは1つずつ。** 「全部まとめて見る」指示は網羅を崩す。
- **列挙してから判定する。** 「catch を全て挙げよ → 1件ずつ失敗時の見え方を答えよ」のように
  作業を分解する。列挙の物量が多い場合はファイル/モジュール単位に分割してよいが、省略しない。
- **契約突合は必ず両側を開く。** 片側だけ読んで「たぶん合っている」は禁止。
- **全CRUD動詞・全コントローラ・全ルートを機械的に**巡回する（sibling 取りこぼし対策）。
- 指摘の物量より**裏取りの質**を優先する。失敗シナリオが書けない指摘は報告しない。
- 監査ステップは reasoning effort を高めに。単純な1パスで終えない。

## やってはいけないこと
- 監査中に**コードを修正しない**（発見と修正を混ぜない。修正は登録後に別ブランチで）。
- 未検証の指摘を高深刻度として断定しない（敵対的再検証を通す）。
- 上限（top-N・サンプリング・無再試行）で打ち切ったのに「網羅した」と述べない。**打ち切ったら明記**。
- 「使われていない前提」を推測で断定しない。撤去提案の前に未使用を grep で確認する。
