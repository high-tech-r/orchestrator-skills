# Skill: design（設計書生成）

## いつ使うか
- 機能の設計書を生成するとき
- Gate 1で差し戻されて設計を修正するとき

## 入力
- 対象機能の要件（バックログの1アイテム）
- 技術スタック情報
- 既存システム情報（追加開発の場合）
- 差し戻し時: feedback_history（過去の指摘事項）

## 出力ファイル
`docs/design/F-XXX.md` に以下の構造で保存する。

```markdown
# 設計書: F-XXX（機能名）

## 機能概要
- サマリー: ...
- アクター: ...
- 事前条件: ...
- 事後条件: ...

## メインフロー
1. ...
2. ...

## エラーフロー
| 条件 | 処理 | HTTPステータス |
|------|------|---------------|
| ... | ... | ... |

### エラーカテゴリチェック
- [x] バリデーションエラー
- [x] ビジネスロジックエラー
- [ ] 認証・認可エラー（該当なしの場合は理由を記載）
- [x] 外部サービス障害
- [x] インフラ障害（DB・ネットワーク）

## インターフェース仕様
機能の**種別**を選び、その枠で書く（HTTP前提で空振りさせない）。1機能が複数種別を持つこともある。

### 種別: HTTP API（該当する場合）
#### POST /api/xxx
- リクエスト: ...
- レスポンス（成功）: ...
- レスポンス（エラー）: ...

### 種別: 非同期ジョブ / キュー（該当する場合）
- ジョブ名 / 投入トリガ: ...
- ペイロード（入力）: ...
- 処理内容と完了条件: ...
- 失敗時の挙動（リトライ・冪等性・デッドレター）: ...

### 種別: イベント / リスナー（該当する場合）
- イベント名 / 発火条件: ...
- リスナーの処理: ...

### 種別: CLIコマンド（該当する場合）
- コマンド / 引数: ...
- 入出力・副作用: ...

### 種別: スケジュール（該当する場合）
- 実行頻度 / トリガ: ...
- 処理内容: ...

> 起動・動作確認・テストの具体は**アクティブなスタックプロファイル**
> （`.orchestrator/stack-profile.md`）に従う。設計はインターフェースの契約に集中する。

## DB変更
| テーブル | 操作 | カラム |
|---------|------|--------|
| ... | CREATE/ALTER | ... |

## 受入条件トレーサビリティ
| 受入条件 | カバー箇所 |
|---------|-----------|
| "条件1の文言" | メインフロー Step 2, インターフェース仕様 |
| "条件2の文言" | エラーフロー, DB変更 |

## 他機能への影響
- F-YYY: ...（依存関係があれば記載）
```

## 重要なルール
- エラーフローは「バリデーション」「ビジネスロジック」「外部サービス」「インフラ」の4カテゴリを必ず検討する
- 「適宜」「必要に応じて」等の曖昧表現は使わない。具体的に書く
- 受入条件トレーサビリティは全受入条件について記載する。漏れがあればGate 1で弾かれる
- 差し戻し時はfeedback_historyの過去の指摘を確認し、同じ問題を繰り返さない
