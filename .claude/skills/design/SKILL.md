# Skill: design（設計書生成）

## いつ使うか
- 機能の設計書を生成するとき
- Gate 1で差し戻されて設計を修正するとき

## 入力
- 対象機能の要件（バックログの1アイテム）
- 技術スタック情報
- 既存システム情報（追加開発の場合）
- 差し戻し時: feedback_history（過去の指摘事項）

## 出力ファイル
`docs/design/F-XXX.md` に以下の構造で保存する。

```markdown
# 設計書: F-XXX（機能名）

## 機能概要
- サマリー: ...
- アクター: ...
- 事前条件: ...
- 事後条件: ...

## メインフロー
1. ...
2. ...

## エラーフロー
| 条件 | 処理 | HTTPステータス |
|------|------|---------------|
| ... | ... | ... |

### エラーカテゴリチェック
- [x] バリデーションエラー
- [x] ビジネスロジックエラー
- [ ] 認証・認可エラー（該当なしの場合は理由を記載）
- [x] 外部サービス障害
- [x] インフラ障害（DB・ネットワーク）

## API仕様
### POST /api/xxx
- リクエスト: ...
- レスポンス（成功）: ...
- レスポンス（エラー）: ...

## DB変更
| テーブル | 操作 | カラム |
|---------|------|--------|
| ... | CREATE/ALTER | ... |

## 受入条件トレーサビリティ
| 受入条件 | カバー箇所 |
|---------|-----------|
| "条件1の文言" | メインフロー Step 2, API仕様 |
| "条件2の文言" | エラーフロー, DB変更 |

## 他機能への影響
- F-YYY: ...（依存関係があれば記載）
```

## 重要なルール
- エラーフローは「バリデーション」「ビジネスロジック」「外部サービス」「インフラ」の4カテゴリを必ず検討する
- 「適宜」「必要に応じて」等の曖昧表現は使わない。具体的に書く
- 受入条件トレーサビリティは全受入条件について記載する。漏れがあればGate 1で弾かれる
- 差し戻し時はfeedback_historyの過去の指摘を確認し、同じ問題を繰り返さない
